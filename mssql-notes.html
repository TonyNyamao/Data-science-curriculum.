Introduction & Setup

Basic

What is Microsoft SQL Server?

SQL Server is Microsoft's relational database management system (RDBMS). It uses Transact-SQL (T-SQL) — an extension of standard SQL with procedural constructs, error handling, and built-in functions specific to SQL Server.

SSMSSQL Server Management Studio — the main GUI tool for writing queries, managing databases, and viewing execution plans. T-SQLTransact-SQL — Microsoft's SQL dialect. Adds IF/ELSE, WHILE loops, TRY/CATCH, variables, and proprietary functions on top of ANSI SQL. InstanceA single running installation of SQL Server. One machine can host multiple instances. DatabaseA container for tables, views, procedures, and other objects. One instance holds many databases.

Essential first commands

T-SQL basics
-- Switch to a specific database
USE my_database;

-- List all databases on the instance
SELECT name FROM sys.databases;

-- List all tables in current database
SELECT TABLE_NAME
FROM INFORMATION_SCHEMA.TABLES
WHERE TABLE_TYPE = 'BASE TABLE';

-- Check current database
SELECT DB_NAME();

-- Check SQL Server version
SELECT @@VERSION;
ℹ T-SQL statements end with a semicolon ; — it's optional in most cases but is best practice and required inside CTEs.

DDL — Create & Manage Tables

Basic

CREATE TABLE

Define the structure of a table — columns, data types, and constraints.

CREATE TABLE with constraints
CREATE TABLE customers (
    id          INT            IDENTITY(1,1) PRIMARY KEY,
    name        NVARCHAR(100)  NOT NULL,
    email       VARCHAR(150)   NOT NULL UNIQUE,
    city        VARCHAR(80)    NULL,
    created_at  DATETIME2      NOT NULL DEFAULT GETDATE(),
    is_active   BIT            NOT NULL DEFAULT 1
);

CREATE TABLE orders (
    id          INT             IDENTITY(1,1) PRIMARY KEY,
    customer_id INT             NOT NULL,
    order_date  DATETIME2       NOT NULL DEFAULT GETDATE(),
    status      VARCHAR(20)     NOT NULL
                CONSTRAINT chk_status
                CHECK (status IN ('pending','processing','completed','cancelled')),
    total       DECIMAL(12,2)  NOT NULL DEFAULT 0,
    CONSTRAINT fk_orders_customer
        FOREIGN KEY (customer_id) REFERENCES customers(id)
);

ALTER & DROP

Modifying tables
-- Add a column
ALTER TABLE customers ADD phone VARCHAR(20) NULL;

-- Change a column's data type
ALTER TABLE customers ALTER COLUMN phone NVARCHAR(30) NULL;

-- Drop a column
ALTER TABLE customers DROP COLUMN phone;

-- Rename a table (T-SQL system proc)
EXEC sp_rename 'customers', 'clients';

-- Rename a column
EXEC sp_rename 'customers.name', 'full_name', 'COLUMN';

-- Drop a table entirely
DROP TABLE IF EXISTS customers;

Constraints summary

FOREIGN KEYEnforces referential integrity between tables.
UNIQUEEnsures all values in a column are distinct.
NOT NULLPrevents NULL values in a column.

DML — Insert, Update, Delete

Basic

INSERT

Inserting rows
-- Single row
INSERT INTO customers (name, email, city)
VALUES ('Alice Mwangi', '[email protected]', 'Nairobi');

-- Multiple rows
INSERT INTO customers (name, email, city)
VALUES
    ('Bob Otieno',   '[email protected]',  'Mombasa'),
    ('Carol Njeri',  '[email protected]','Kisumu');

-- Insert from SELECT (copy data between tables)
INSERT INTO customers_archive (name, email, city, created_at)
SELECT name, email, city, created_at
FROM customers
WHERE is_active = 0;

-- Capture the new IDENTITY value
INSERT INTO customers (name, email)
VALUES ('Dave Kamau', '[email protected]');
SELECT SCOPE_IDENTITY() AS new_id;

UPDATE

Updating rows
-- Basic update
UPDATE customers
SET   city = 'Eldoret'
WHERE id = 5;

-- Update multiple columns
UPDATE customers
SET   city      = 'Nakuru',
      is_active = 1
WHERE email LIKE '%gmail.com';

-- Update with JOIN (update from another table's data)
UPDATE c
SET   c.city = a.new_city
FROM  customers c
JOIN  city_updates a ON c.id = a.customer_id;

DELETE & TRUNCATE

Removing rows
-- Delete specific rows (logged, can be rolled back)
DELETE FROM customers WHERE is_active = 0;

-- Delete all rows but keep table structure
-- (minimally logged, resets IDENTITY, faster than DELETE)
TRUNCATE TABLE customers;

-- Safe pattern: check before deleting
SELECT * FROM customers WHERE is_active = 0; -- preview first
DELETE FROM customers WHERE is_active = 0; -- then delete

MERGE (Upsert)

Update existing rows and insert new ones in a single statement — the T-SQL "upsert" pattern.

MERGE statement
MERGE INTO customers AS target
USING staging_customers AS source
    ON target.email = source.email
WHEN MATCHED THEN
    UPDATE SET
        target.name = source.name,
        target.city = source.city
WHEN NOT MATCHED BY TARGET THEN
    INSERT (name, email, city)
    VALUES (source.name, source.email, source.city)
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;  -- optional: remove rows not in source

SELECT & Filtering

Basic

SELECT clause order

T-SQL evaluates clauses in this logical order — understanding this prevents many confusion points:

1. FROMIdentify source tables / joins 2. WHEREFilter individual rows 3. GROUP BYGroup rows into buckets 4. HAVINGFilter groups (not rows) 5. SELECTChoose columns / expressions 6. DISTINCTRemove duplicate rows 7. ORDER BYSort the result set 8. TOP / OFFSETLimit rows returned

Filtering & predicates

WHERE clause patterns
-- Comparison operators
SELECT * FROM products WHERE price > 500;
SELECT * FROM products WHERE price BETWEEN 100 AND 500;

-- IN list
SELECT * FROM orders WHERE status IN ('pending', 'processing');

-- Pattern matching with LIKE
-- %  = any sequence of chars     _ = exactly one char
SELECT * FROM customers WHERE email LIKE '%@gmail.com';
SELECT * FROM customers WHERE name  LIKE 'A%';

-- NULL checks (never use = NULL)
SELECT * FROM customers WHERE city IS NULL;
SELECT * FROM customers WHERE city IS NOT NULL;

-- Date range
SELECT * FROM orders
WHERE  order_date >= '2024-01-01'
  AND  order_date  < '2025-01-01';

-- TOP N rows
SELECT TOP 10 * FROM orders ORDER BY order_date DESC;

-- Pagination with OFFSET / FETCH
SELECT * FROM products
ORDER BY name
OFFSET  20 ROWS
FETCH NEXT 10 ROWS ONLY; -- page 3 of 10-per-page

Useful built-in functions

String, date & conversion functions
-- String functions
LEN('hello')                       -- 5
UPPER('hello')                     -- 'HELLO'
LOWER('HELLO')                     -- 'hello'
LTRIM(RTRIM('  hi  '))            -- 'hi' (trim whitespace)
TRIM('  hi  ')                     -- 'hi' (SQL Server 2017+)
SUBSTRING('Nairobi', 1, 3)         -- 'Nai'
CHARINDEX('@', '[email protected]')    -- 5
REPLACE('Hello World','World','SQL') -- 'Hello SQL'
CONCAT('Hello', ' ', 'World')     -- 'Hello World'
FORMAT(1234567, 'N0')              -- '1,234,567'

-- Date functions
GETDATE()                         -- current datetime
GETUTCDATE()                      -- current UTC datetime
SYSDATETIME()                     -- higher precision
YEAR(order_date)                   -- extract year
MONTH(order_date)                  -- extract month
DAY(order_date)                    -- extract day
DATEPART(WEEKDAY, order_date)      -- day of week (1=Sun)
DATEDIFF(DAY, start_date, end_date) -- difference in days
DATEADD(DAY, 30, order_date)       -- add 30 days
FORMAT(order_date, 'yyyy-MM')     -- '2024-06'

-- Conversion
CAST('123' AS INT)                -- 123
CONVERT(VARCHAR(10), GETDATE(), 103)  -- 'DD/MM/YYYY'
TRY_CAST('abc' AS INT)           -- NULL (safe, no error)
ISNULL(city, 'Unknown')          -- replace NULL
COALESCE(phone, email, 'N/A')    -- first non-NULL

JOINs

Intermediate

JOIN types

JOIN TypeReturns
INNER JOINOnly rows with a match in BOTH tables.
LEFT JOINAll rows from left table; NULLs where no match in right.
RIGHT JOINAll rows from right table; NULLs where no match in left.
FULL OUTER JOINAll rows from both tables; NULLs where no match on either side.
CROSS JOINCartesian product — every row × every row. Use carefully.
SELF JOINTable joined to itself using aliases. Good for hierarchies.

JOIN examples

JOIN patterns
-- INNER JOIN: orders with customer names
SELECT o.id, c.name, o.order_date, o.total
FROM   orders o
INNER JOIN customers c ON o.customer_id = c.id;

-- LEFT JOIN: all customers, even those with no orders
SELECT c.name, o.id AS order_id
FROM   customers c
LEFT JOIN orders o ON c.id = o.customer_id;

-- Find customers with NO orders (anti-join pattern)
SELECT c.name
FROM   customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE  o.id IS NULL;

-- Three-table JOIN
SELECT c.name, p.name AS product, oi.quantity
FROM   orders o
JOIN   customers c   ON o.customer_id  = c.id
JOIN   order_items oi ON o.id          = oi.order_id
JOIN   products p    ON oi.product_id  = p.id;

-- SELF JOIN: employee reporting hierarchy
SELECT e.name AS employee,
       m.name AS manager
FROM   employees e
LEFT JOIN employees m ON e.manager_id = m.id;

Aggregation & Grouping

Intermediate

Aggregate functions

Aggregates
SELECT
    COUNT(*)                  AS total_rows,
    COUNT(city)               AS non_null_cities,  -- NULLs ignored
    COUNT(DISTINCT city)      AS unique_cities,
    SUM(total)                AS revenue,
    AVG(total)                AS avg_order,
    MIN(order_date)           AS first_order,
    MAX(order_date)           AS last_order,
    STDEV(total)              AS std_dev_total
FROM orders;

-- GROUP BY with HAVING
SELECT   city,
         COUNT(*) AS customer_count,
         AVG(lifetime_spend) AS avg_spend
FROM     customers
GROUP BY city
HAVING   COUNT(*) > 5       -- filter AFTER grouping
ORDER BY customer_count DESC;

-- ROLLUP: subtotals + grand total
SELECT   ISNULL(category, 'TOTAL') AS category,
         SUM(price) AS revenue
FROM     products
GROUP BY ROLLUP(category);
✓ WHERE vs HAVING: Use WHERE to filter rows before grouping. Use HAVING to filter groups after aggregation. You cannot use aggregate functions in WHERE.

Subqueries & CTEs

Intermediate

Subqueries

Subquery patterns
-- In WHERE (scalar subquery)
SELECT name, price
FROM   products
WHERE  price > (SELECT AVG(price) FROM products);

-- IN subquery
SELECT name FROM customers
WHERE  id IN (SELECT customer_id FROM orders
               WHERE total > 10000);

-- EXISTS (often faster than IN for large sets)
SELECT name FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.id
      AND o.total > 5000
);

-- Derived table (subquery in FROM)
SELECT city, avg_spend
FROM (
    SELECT city, AVG(total) AS avg_spend
    FROM   orders o
    JOIN   customers c ON o.customer_id = c.id
    GROUP BY city
) AS city_stats
WHERE avg_spend > 2000;

CTEs (Common Table Expressions)

CTEs make complex queries readable by naming intermediate result sets. They do not persist — they exist only for the duration of the query.

CTE patterns
-- Single CTE
WITH high_value_orders AS (
    SELECT customer_id, SUM(total) AS spend
    FROM   orders
    GROUP BY customer_id
    HAVING SUM(total) > 50000
)
SELECT c.name, hvo.spend
FROM   customers c
JOIN   high_value_orders hvo ON c.id = hvo.customer_id;

-- Multiple CTEs (chained with comma)
WITH
monthly_revenue AS (
    SELECT FORMAT(order_date, 'yyyy-MM') AS mo,
           SUM(total) AS revenue
    FROM   orders
    GROUP BY FORMAT(order_date, 'yyyy-MM')
),
ranked AS (
    SELECT mo, revenue,
           RANK() OVER (ORDER BY revenue DESC) AS rnk
    FROM   monthly_revenue
)
SELECT * FROM ranked WHERE rnk <= 3;

-- Recursive CTE: employee hierarchy
WITH org_chart AS (
    -- Anchor: top-level (CEO)
    SELECT id, name, manager_id, 0 AS level
    FROM   employees WHERE manager_id IS NULL
    UNION ALL
    -- Recursive: employees reporting to the anchor
    SELECT e.id, e.name, e.manager_id, oc.level + 1
    FROM   employees e
    JOIN   org_chart oc ON e.manager_id = oc.id
)
SELECT REPLICATE('  ', level) + name AS hierarchy
FROM   org_chart;

Indexes

Intermediate

Index types

TypeNotes
ClusteredPhysically sorts the table by the index key. One per table. Auto-created on PRIMARY KEY.
Non-ClusteredSeparate structure pointing to table rows. Many per table. Most common type.
UniqueEnforces uniqueness + creates an index. Faster lookups on unique columns.
CoveringNon-clustered index that includes all columns needed by a query — avoids table lookup entirely.
FilteredIndex on a subset of rows (WHERE is_active = 1). Smaller, faster for selective queries.
ColumnstoreColumn-oriented storage. Excellent for analytics/reporting queries over large tables.

Creating & managing indexes

Index operations
-- Basic non-clustered index
CREATE INDEX ix_orders_customer
    ON orders(customer_id);

-- Composite index (order matters: most selective first)
CREATE INDEX ix_orders_status_date
    ON orders(status, order_date);

-- Covering index (INCLUDE non-key columns)
CREATE NONCLUSTERED INDEX ix_orders_covering
    ON orders(customer_id)
    INCLUDE (order_date, total);

-- Filtered index (only active records)
CREATE INDEX ix_customers_active
    ON customers(email)
    WHERE is_active = 1;

-- Unique index
CREATE UNIQUE INDEX uix_customers_email
    ON customers(email);

-- View existing indexes
SELECT * FROM sys.indexes
WHERE object_id = OBJECT_ID('orders');

-- Drop an index
DROP INDEX ix_orders_customer ON orders;
⚠ Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE). Index columns used in WHERE, JOIN ON, and ORDER BY — but don't over-index OLTP tables with heavy write loads.

Views & Synonyms

Intermediate

Views

A view is a saved SELECT query. It doesn't store data (unless indexed). Use views to simplify complex queries, restrict column access, and create stable interfaces over changing tables.

Creating views
-- Create a view
CREATE VIEW vw_order_summary AS
SELECT
    o.id          AS order_id,
    c.name        AS customer_name,
    c.city,
    o.order_date,
    o.status,
    o.total
FROM  orders o
JOIN  customers c ON o.customer_id = c.id;

-- Query it like a table
SELECT * FROM vw_order_summary
WHERE city = 'Nairobi' AND status = 'completed';

-- Update a view definition
ALTER VIEW vw_order_summary AS
SELECT /* updated query */ ...;

-- Drop a view
DROP VIEW IF EXISTS vw_order_summary;

-- Indexed (materialised) view — stores the data physically
CREATE VIEW vw_product_revenue WITH SCHEMABINDING AS
SELECT product_id, SUM(quantity * unit_price) AS revenue,
       COUNT_BIG(*) AS cnt
FROM   dbo.order_items
GROUP BY product_id;

CREATE UNIQUE CLUSTERED INDEX ix_vw_product_revenue
    ON vw_product_revenue(product_id); -- now materialised

Window Functions

Advanced

Syntax & concepts

Window functions perform calculations across a set of rows related to the current row — without collapsing them like GROUP BY does.

Window function syntax
function_name() OVER (
    [PARTITION BY column]    -- reset window per group
    [ORDER BY column]        -- define row order within window
    [ROWS/RANGE BETWEEN ...] -- define frame size
)

Ranking functions

ROW_NUMBER, RANK, DENSE_RANK, NTILE
SELECT
    name, city, total_spend,
    -- Unique sequential number (no ties)
    ROW_NUMBER()   OVER (PARTITION BY city ORDER BY total_spend DESC) AS row_num,
    -- Rank with gaps on ties (1,1,3,4)
    RANK()         OVER (PARTITION BY city ORDER BY total_spend DESC) AS rnk,
    -- Rank without gaps on ties (1,1,2,3)
    DENSE_RANK()   OVER (PARTITION BY city ORDER BY total_spend DESC) AS dense_rnk,
    -- Divide into N equal buckets
    NTILE(4)       OVER (ORDER BY total_spend DESC)                    AS quartile
FROM customer_spend;

-- Top N per group using ROW_NUMBER
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY city ORDER BY total_spend DESC) AS rn
    FROM customer_spend
)
SELECT * FROM ranked WHERE rn <= 3; -- top 3 per city

Analytic & offset functions

LAG, LEAD, FIRST_VALUE, running totals
SELECT
    mo,
    revenue,
    -- Previous row's value
    LAG(revenue, 1, 0) OVER (ORDER BY mo) AS prev_month,
    -- Next row's value
    LEAD(revenue, 1, 0) OVER (ORDER BY mo) AS next_month,
    -- Month-over-month % change
    ROUND((revenue - LAG(revenue) OVER (ORDER BY mo))
           / NULLIF(LAG(revenue) OVER (ORDER BY mo), 0) * 100, 1) AS pct_change,
    -- Cumulative (running) total
    SUM(revenue) OVER (ORDER BY mo
                       ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW)
                  AS running_total,
    -- 3-month moving average
    AVG(revenue) OVER (ORDER BY mo
                       ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)
                  AS moving_avg_3m
FROM monthly_revenue;

Stored Procedures

Advanced

Creating stored procedures

Stored procedures are pre-compiled T-SQL programs stored in the database. They accept parameters, contain logic, and can return result sets or output values.

Stored procedure with error handling
CREATE PROCEDURE usp_GetCustomerOrders
    @customer_id  INT,
    @status       VARCHAR(20) = NULL  -- optional parameter
AS
BEGIN
    SET NOCOUNT ON;  -- suppress "N rows affected" messages

    BEGIN TRY
        SELECT
            o.id, o.order_date, o.status, o.total
        FROM   orders o
        WHERE  o.customer_id = @customer_id
          AND  (@status IS NULL OR o.status = @status)
        ORDER BY o.order_date DESC;
    END TRY
    BEGIN CATCH
        SELECT
            ERROR_NUMBER()    AS error_number,
            ERROR_MESSAGE()   AS error_message,
            ERROR_LINE()      AS error_line;
    END CATCH
END;

-- Execute it
EXEC usp_GetCustomerOrders @customer_id = 42;
EXEC usp_GetCustomerOrders @customer_id = 42, @status = 'completed';

Variables, conditionals & loops

T-SQL control flow
DECLARE @total_orders INT;
DECLARE @msg          NVARCHAR(200);

-- Assign a variable
SELECT @total_orders = COUNT(*) FROM orders;
-- Or: SET @total_orders = 100;

-- IF / ELSE
IF @total_orders > 1000
    SET @msg = 'High volume';
ELSE IF @total_orders > 100
    SET @msg = 'Medium volume';
ELSE
    SET @msg = 'Low volume';

PRINT @msg;

-- WHILE loop
DECLARE @i INT = 1;
WHILE @i <= 5
BEGIN
    PRINT 'Row: ' + CAST(@i AS VARCHAR);
    SET @i = @i + 1;
END;

-- CASE expression
SELECT name,
    CASE
        WHEN total > 10000 THEN 'Platinum'
        WHEN total >  5000 THEN 'Gold'
        WHEN total >  1000 THEN 'Silver'
        ELSE                    'Bronze'
    END AS tier
FROM customer_spend;

Triggers

Advanced

DML Triggers

Triggers fire automatically when a DML event (INSERT, UPDATE, DELETE) occurs. They use the virtual tables INSERTED and DELETED to access changed data.

Audit trigger example
-- Audit table to log changes
CREATE TABLE orders_audit (
    audit_id    INT IDENTITY PRIMARY KEY,
    order_id    INT,
    action      VARCHAR(10),
    old_status  VARCHAR(20),
    new_status  VARCHAR(20),
    changed_at  DATETIME2 DEFAULT GETDATE(),
    changed_by  NVARCHAR(100) DEFAULT SYSTEM_USER
);

-- AFTER UPDATE trigger
CREATE TRIGGER trg_orders_audit
ON orders
AFTER UPDATE
AS
BEGIN
    SET NOCOUNT ON;
    INSERT INTO orders_audit (order_id, action, old_status, new_status)
    SELECT
        d.id,
        'UPDATE',
        d.status,   -- DELETED = before state
        i.status    -- INSERTED = after state
    FROM deleted d
    JOIN inserted i ON d.id = i.id
    WHERE d.status <> i.status;  -- only log actual status changes
END;
⚠ Triggers fire once per statement, not once per row. Use INSERTED and DELETED as sets — never assume they contain a single row.

Transactions & Locking

Advanced

ACID properties

AtomicityAll operations in a transaction succeed together, or none do. ConsistencyThe database moves from one valid state to another. IsolationConcurrent transactions don't interfere with each other. DurabilityCommitted changes survive system failures.

Transaction control

Explicit transactions with error handling
BEGIN TRANSACTION;

BEGIN TRY
    -- Transfer stock between warehouses
    UPDATE inventory
    SET    stock = stock - 10
    WHERE  warehouse_id = 1 AND product_id = 42;

    UPDATE inventory
    SET    stock = stock + 10
    WHERE  warehouse_id = 2 AND product_id = 42;

    -- Both succeeded — commit
    COMMIT TRANSACTION;
    PRINT 'Transfer complete.';
END TRY
BEGIN CATCH
    -- Something failed — undo everything
    ROLLBACK TRANSACTION;
    PRINT 'Transfer failed: ' + ERROR_MESSAGE();
END CATCH;

-- Savepoints (partial rollback)
BEGIN TRANSACTION;
    INSERT INTO log_table VALUES('step 1');
    SAVE TRANSACTION sp1;
    INSERT INTO log_table VALUES('step 2');
    ROLLBACK TRANSACTION sp1; -- undo only step 2
COMMIT TRANSACTION;          -- commits step 1

Isolation levels

LevelDirty ReadNon-rep. ReadPhantom
READ UNCOMMITTEDYesYesYes
READ COMMITTED (default)NoYesYes
REPEATABLE READNoNoYes
SERIALIZABLENoNoNo
SNAPSHOTNoNoNo (uses versioning)
Setting isolation level
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- Or as a query hint
SELECT * FROM orders WITH (NOLOCK);  -- READ UNCOMMITTED hint

Query Optimisation

Advanced

Reading execution plans

Always check the execution plan before optimising. In SSMS: press Ctrl+M for actual plan or Ctrl+L for estimated.

  • Look for Table Scan or Clustered Index Scan — these scan entire tables and are usually the culprit.
  • Prefer Index Seek over Index Scan — seeks jump directly to relevant rows.
  • Fat arrows between operators indicate large data movement — investigate those operators first.
  • Hash Match and Sort operators are expensive — see if an index can eliminate them.
  • Check estimated vs actual row counts — large discrepancies mean stale statistics.

Common performance patterns

Optimisation techniques
-- 1. SARGABLE predicates (let the engine use an index)
-- BAD: function on indexed column prevents index seek
WHERE YEAR(order_date) = 2024

-- GOOD: range predicate is SARGABLE
WHERE order_date >= '2024-01-01' AND order_date < '2025-01-01'

-- 2. Avoid SELECT * in production
-- Fetches unnecessary columns, blocks covering indexes
SELECT id, name, email FROM customers; -- explicit columns

-- 3. Update statistics when queries suddenly slow down
UPDATE STATISTICS orders;
UPDATE STATISTICS orders ix_orders_status_date;

-- 4. sp_WhoIsActive — find blocking queries (free tool)
EXEC sp_WhoIsActive;

-- 5. Find missing indexes (suggested by SQL Server)
SELECT TOP 10
    migs.avg_total_user_cost * migs.avg_user_impact * (migs.user_seeks + migs.user_scans)
        AS improvement_measure,
    mid.statement AS table_name,
    mid.equality_columns, mid.inequality_columns, mid.included_columns
FROM sys.dm_db_missing_index_groups   mig
JOIN sys.dm_db_missing_index_group_stats migs ON mig.index_group_handle = migs.group_handle
JOIN sys.dm_db_missing_index_details  mid  ON mig.index_handle = mid.index_handle
ORDER BY improvement_measure DESC;

-- 6. Find expensive queries from cache
SELECT TOP 10
    qs.total_elapsed_time / qs.execution_count AS avg_elapsed_us,
    qs.execution_count,
    SUBSTRING(qt.text, (qs.statement_start_offset/2)+1,
        ((CASE qs.statement_end_offset WHEN -1 THEN DATALENGTH(qt.text)
          ELSE qs.statement_end_offset END - qs.statement_start_offset)/2)+1
    ) AS query_text
FROM sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY avg_elapsed_us DESC;

JSON & XML

Advanced

JSON (SQL Server 2016+)

JSON operations
-- Export query results as JSON
SELECT id, name, email
FROM   customers
FOR JSON AUTO;
-- Returns: [{"id":1,"name":"Alice","email":"alice@..."},...]

-- FOR JSON PATH gives more control
SELECT o.id AS 'order.id', c.name AS 'customer.name'
FROM orders o JOIN customers c ON o.customer_id = c.id
FOR JSON PATH;

-- Parse JSON into rows
DECLARE @json NVARCHAR(MAX) = '[
  {"id":1,"name":"Alice","city":"Nairobi"},
  {"id":2,"name":"Bob",  "city":"Mombasa"}
]';

SELECT *
FROM OPENJSON(@json)
WITH (
    id   INT          '$.id',
    name NVARCHAR(100) '$.name',
    city VARCHAR(80)  '$.city'
);

-- Read a JSON property from a column
SELECT JSON_VALUE(metadata, '$.source') AS source
FROM   orders;

-- Update a JSON property
UPDATE orders
SET    metadata = JSON_MODIFY(metadata, '$.source', 'web')
WHERE  id = 1;

Logins, users & roles

User management
-- Create a server login
CREATE LOGIN app_user WITH PASSWORD = 'Str0ng!Pass';

-- Create a database user mapped to the login
USE my_database;
CREATE USER app_user FOR LOGIN app_user;

-- Add to a built-in role
ALTER ROLE db_datareader ADD MEMBER app_user;  -- read only
ALTER ROLE db_datawriter ADD MEMBER app_user;  -- read + write

-- Granular permissions
GRANT SELECT ON customers  TO app_user;
GRANT SELECT, INSERT ON orders TO app_user;
DENY  DELETE ON orders     TO app_user;   -- explicit deny always wins

-- Grant execute on a procedure (preferred pattern)
GRANT EXECUTE ON usp_GetCustomerOrders TO app_user;

-- Revoke a permission
REVOKE SELECT ON customers FROM app_user;

Row-Level Security (RLS)

RLS restricts which rows a user can see — transparently applied to every query.

Row-Level Security setup
-- Predicate function: each user sees only their region's data
CREATE FUNCTION dbo.fn_rls_filter (@region VARCHAR(50))
RETURNS TABLE WITH SCHEMABINDING AS
RETURN
    SELECT 1 AS result
    WHERE  @region = USER_NAME()
        OR USER_NAME() = 'admin';

-- Security policy applying the filter
CREATE SECURITY POLICY rls_region_policy
ADD FILTER PREDICATE dbo.fn_rls_filter(region)
ON dbo.sales
WITH (STATE = ON);
✓ Best practice: never give application users direct table access. Instead, grant EXECUTE on stored procedures only — this enforces a clean API boundary and makes SQL injection much harder.
